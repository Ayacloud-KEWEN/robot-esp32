# 02 · 小智固件架构解析

> 本篇目标：读懂 `xiaozhi-esp32/` 固件源码的整体结构，理解"一句话"在设备端的完整旅程，为定制硬件打基础。

## 1. 源码地图（main/ 目录）

```
xiaozhi-esp32/main/
├── main.cc                    # 入口 app_main()：初始化并启动 Application
├── application.cc/.h          # ★ 核心调度器（单例）：状态机驱动、协调所有模块
├── device_state_machine.cc/.h # 设备状态机（idle/connecting/listening/speaking...）
├── audio/                     # ★ 音频子系统
│   ├── audio_service.cc       #   采集/播放任务、OPUS 编解码调度
│   ├── audio_codec.cc         #   音频硬件抽象（Codec 芯片 / 无 Codec 直连 I2S）
│   ├── codecs/  processors/   #   具体编解码与前处理（AEC 回声消除等）
│   └── wake_word/             #   离线唤醒词（ESP-SR）
├── protocols/                 # ★ 与服务器的通信协议
│   ├── websocket_protocol.cc  #   WebSocket 实现
│   └── mqtt_protocol.cc       #   MQTT+UDP 实现
├── boards/                    # ★ 96 块板子的板级适配（定制硬件主战场）
│   └── common/                #   板级基类：Board / WifiBoard / Ml307Board
├── display/                   # 屏幕与 UI（LVGL）、表情显示
├── led/                       # 指示灯
├── assets/                    # 音效、字体、表情等资源（打包成 assets.bin）
├── mcp_server.cc              # 设备端 MCP 服务器：向 AI 暴露设备控制工具
├── ota.cc                     # OTA 检查升级 + 从服务器获取 WebSocket 配置
└── settings.cc                # NVS 配置读写
```

## 2. 核心设计：状态机 + 事件驱动

固件的骨架是 `Application` 单例 + 设备状态机：

```
[启动] → kDeviceStateStarting → 连Wi-Fi → kDeviceStateIdle（待机，等唤醒）
   唤醒词命中/按键 ↓
kDeviceStateConnecting（连服务器）→ kDeviceStateListening（录音上传）
   服务器回复 ↓                          ↑ 说完一轮
kDeviceStateSpeaking（播放TTS）──────────┘
```

理解要点：
- **一切行为由状态驱动**：比如只有 `Listening` 状态才会把麦克风数据编码上传；`Speaking` 状态收到用户打断（说话/按键）会切回 `Listening`。
- **模块间用事件/回调解耦**：协议层收到服务器 JSON（如 `tts start`），转成事件交给 Application 切换状态。
- 精读顺序建议：`main.cc` → `application.cc`（重点）→ `device_state_machine.cc`。

## 3. 一句话的旅程（设备端视角）

```
麦克风 (I2S) → AudioCodec 采集 PCM → AFE前处理(AEC回声消除/降噪)
  → 唤醒词检测(ESP-SR, 离线)  [Idle 时]
  → OPUS 编码 (60ms/帧)       [Listening 时]
  → Protocol 发送二进制帧 → 服务器
服务器返回 ← JSON控制消息(状态/表情/文本) + OPUS音频帧
  → OPUS 解码 → AudioCodec 播放 (I2S → 功放 → 喇叭)
  → 同时 Display 显示文字/表情
```

配套协议文档：`xiaozhi-esp32/docs/websocket.md`（务必精读——自建服务器就是实现这份协议的另一端）。

关键消息类型（WebSocket）：
- 设备→服务器：`hello`（能力协商）、`listen`（开始/停止收音）、`abort`（打断）、OPUS 二进制帧
- 服务器→设备：`hello`、`stt`（识别文本）、`llm`（表情）、`tts`（start/stop/sentence_start + 音频帧）、MCP 调用

## 4. 板级适配（Board）——定制硬件的关键

每块板子在 `boards/` 下一个目录，核心是一个继承 `WifiBoard`（或 `Ml307Board`）的类，声明：

- 用哪个 **AudioCodec**（什么芯片、I2S/I2C 引脚号）
- 用哪个 **Display**（驱动芯片、SPI 引脚、分辨率）
- **按键**、**LED**、电源管理引脚
- 通过 `DECLARE_BOARD` 宏注册；`menuconfig` 里选择板型即选择编译哪个目录

**定制自己的板子 = 复制一个最接近的现有板目录，改引脚和外设配置。**
建议精读两个例子：一个带屏幕的（如 `lichuang-dev`）和一个极简的（如 `bread-compact-wifi` 面包板）。

## 5. 设备端 MCP：AI 如何控制硬件

`mcp_server.cc` 在设备上实现了一个小型 MCP 服务器，把设备能力注册为"工具"：

```cpp
// 概念示例：注册一个调音量的工具
mcp_server.AddTool("self.audio_speaker.set_volume",
                   "设置音量 0-100", ...,
                   [](args) { codec->SetOutputVolume(args["volume"]); });
```

对话时大模型看到工具列表，说"声音小一点"→ LLM 生成工具调用 → 经服务器转发到设备执行。
**给陪伴机器人加电机、灯光、传感器，就是在板级代码里注册新的 MCP 工具。**

## 6. 动手实验（过关标准）

1. **编译烧录原版固件**，配网后连虾哥官方服务器或自建服务器对话成功。
2. **加日志观察状态机**：在 `device_state_machine.cc` 状态切换处加 `ESP_LOGI`，对着 monitor 说一轮话，写出完整状态序列。
3. **改一个 MCP 工具**：在 `mcp_server.cc` 里加一个 `self.test.hello` 工具（返回一句话），对机器人说"调用测试工具"验证 LLM 能发现并调用它。
4. **读懂一个板级目录**：任选一块板，画出它的引脚连接表（哪个 GPIO 接麦克风/功放/屏幕/按键）。
