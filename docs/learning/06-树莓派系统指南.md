## 2. 理解这台机器的"四层管理体系"

初学者最容易晕的是：同一台机器上，程序被四种不同的方式管理着。查问题时先想清楚目标属于哪一层：

```
① systemd（系统服务层）：Linux 自带的服务管家，开机自启的底层服务
     → frpc、nginx、docker、llama、go2rtc、tailscale、samba、pm2...
② Docker（容器层）：隔离打包的应用，大部分应用在这里
     → 小智全家桶、Home Assistant、Jellyfin、Portainer...
③ CasaOS（面板层）：网页版"应用商店"，它管理的其实也是 Docker 容器
     → 浏览器访问 http://192.168.1.29 就是它（80 端口）
④ pm2（Node 进程层）：Node.js 应用的进程管家
     → ayashop-front、strapi-shop
```

对应的"查状态"命令：

```bash
systemctl status frpc          # ① systemd 服务
docker ps                      # ② 容器列表
# ③ CasaOS 用浏览器看
pm2 list                       # ④ Node 进程
```

## 5. 以后部署新软件的建议（重要经验）

1. **优先用 Docker Compose，别裸装**。一个应用一个目录一个 `docker-compose.yml`，好装好卸好迁移。裸装的东西（如这台机器上的 nginx、frp、llama）时间久了自己都忘了怎么配的——你现在就深有体会 😄
2. **先查端口再部署**：`sudo ss -tlnp | grep 端口号`。这台机器端口已经很挤了（8000 被 Portainer 占导致小智被迫用 8005 就是教训）。建议新应用从 **18xxx** 段开始用。
3. **密钥永远走 `.env` 文件**（chmod 600），compose 里用 `${变量}` 引用——我们在 `/opt/xiaozhi-server/.env` 已经做了示范。
4. **大文件放 SSD**：媒体、模型、备份都放 `/mnt/ssd`（还有 715G 空闲），别塞 58G 的系统盘。
5. **部署完记录到 git 仓库**：像 `deploy/pi/` 那样把 compose 文件收编进仓库，写清楚端口和用途。三个月后的你会感谢现在的你。
6. **远程管理优先 Tailscale，公开服务才用 frp**：Tailscale 只有你自己的设备能访问（安全）；frp 是真公网暴露（要考虑认证）。管理面板类（8002、Portainer、CasaOS）建议只走 Tailscale。
7. **改配置前先备份**：`sudo cp xxx.yml xxx.yml.bak`，出问题一秒回滚。
8. **内存预算意识**：16G 中已用约 10G。大头：小智 ASR 2.5G、llama 2.45G、chromium 1.2G、HA+Jellyfin 各 0.6G。再上大内存应用前先算账。

## 6. 专题：怎么把大文件放到 SSD（1TB）

### 6.1 基本认识

- 系统盘只有 58G（microSD/NVMe 系统盘），**寿命和空间都金贵**；SSD 有 1TB（`/dev/sda1`），挂载在 `/mnt/ssd`，读写大文件也更快。
- "挂载"的意思：把这块盘接到目录树的某个位置。开机自动挂载由 `/etc/fstab` 负责（已配置好，不用动）。
- 原则：**系统和程序放系统盘，数据放 SSD**。

### 6.2 日常怎么放文件进去

```bash
# 方式一：SSH 命令行（在树莓派上移动/复制）
mv ~/大文件.zip /mnt/ssd/          # 移动
cp -r ~/照片目录 /mnt/ssd/photo/   # 复制目录

# 方式二：从 Windows 传文件过去（在 Windows 上执行）
scp D:\video.mp4 fdcaptain@192.168.1.29:/mnt/ssd/videos/

# 方式三：网上邻居（最方便，适合日常拖拽）
# 树莓派开了 Samba，Windows 资源管理器地址栏输入 \\192.168.1.29 即可像 U 盘一样用

# 方式四：网页上传：filebrowser  http://192.168.1.29:8082
```

现有分类目录：`music / videos / shows / photo / ebook / docker / ollama_models`，新类型文件建议先 `mkdir /mnt/ssd/新分类` 再放。

### 6.3 部署应用时怎么把数据放 SSD（重要习惯）

Docker 应用的数据目录用 volume 映射指到 SSD，例如 compose 里：

```yaml
volumes:
  - /mnt/ssd/jellyfin-config:/config     # 应用数据 → SSD
  - /mnt/ssd/videos:/media               # 媒体库 → SSD
```

小智的音乐目录就是这么做的（`/mnt/ssd/music` 映射进了容器）。以后新应用照抄这个模式。

### 6.4 已经装在系统盘的东西怎么搬（通用三步法）

```bash
sudo systemctl stop 服务名          # ① 先停服务（或 docker stop）
sudo mv /原路径/数据目录 /mnt/ssd/  # ② 搬过去
sudo ln -s /mnt/ssd/数据目录 /原路径/数据目录   # ③ 原地留一个"软链接"（快捷方式），程序无感知
# 重启服务验证正常后才算完成
```

### 6.5 检查空间

```bash
df -h /mnt/ssd                  # SSD 还剩多少
du -sh /mnt/ssd/* | sort -hr    # SSD 里谁占得多
sudo du -xsh /* 2>/dev/null | sort -hr | head   # 系统盘里谁占得多（-x 不跨盘）
```

## 7. 专题：frp 与 Tailscale 的区别和使用建议

两者都解决同一个问题——**家里没有公网 IP，出门怎么访问树莓派**——但思路完全相反：

| | **frp（内网穿透）** | **Tailscale（组网 VPN）** |
|---|---|---|
| 原理 | 树莓派主动连你的 OVH VPS，把端口"搬"到 VPS 的公网 IP 上 | 你的所有设备（树莓派/台式机/iPhone/MacBook）组成一张加密虚拟局域网 |
| 谁能访问 | **全世界任何人**（拿到地址就能连） | **只有登录了你 Tailscale 账号的设备** |
| 安全性 | 暴露面大，被扫描是常态，每个服务必须自带认证 | 端到端加密（WireGuard），外人根本"看不见"这些端口 |
| 依赖 | 需要一台有公网 IP 的 VPS（月租） | 免费额度足够个人用，不需要 VPS |
| 速度 | 全部流量绕经 VPS 中转 | 大多数情况设备间直连（打洞成功时），更快 |
| 客户端 | 访问方不用装任何东西 | 每台访问设备要装 Tailscale 并登录 |
| 典型用途 | 给"别人"用的公开服务 | 给"自己"用的私人访问 |

**判断口诀：这个服务是给谁用的？**
- 给**自己/家人**用（管理面板、文件、远程桌面）→ **Tailscale**
- 给**陌生设备/公众**用（对外网站、要分享的链接、不方便装客户端的设备）→ **frp**

**对这台机器的具体建议**：

| 服务 | 现状 | 建议 |
|---|---|---|
| 智控台 8002 | frp 公网 | **改走 Tailscale**（`http://100.74.246.122:8002`），frp 撤下——管理面板不该公网裸奔 |
| 小智 WebSocket 8005 | frp 公网 | **保留 frp**——机器人硬件出门连 Wi-Fi 就能用，装不了 Tailscale 客户端；已有设备认证兜底 |
| 视觉接口 8003 | frp 公网 | 与 8005 同理保留 |
| 旧版页面 3001/3002 | frp 公网 | 随旧版下线一起撤掉 |
| Portainer/CasaOS/filebrowser 等面板 | 未走 frp ✅ | 保持现状，出门要用就走 Tailscale |

> 附：出门用 Tailscale 访问树莓派 = 手机/电脑上打开 Tailscale 客户端 → 直接访问 `100.74.246.122`（或主机名 `kevin`）上的任意端口，体验和在家一模一样。
