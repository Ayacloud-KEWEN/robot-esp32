# 04 · 大模型与语音 AI 流水线

> 本篇目标：理解语音对话 AI 的完整流水线（VAD→ASR→LLM→TTS），掌握提示词、记忆、工具调用（MCP）等大模型应用核心技术，并能在 xiaozhi-server 中动手调优。

## 1. 语音对话流水线总览

服务器收到设备的 OPUS 音频后，经过这条流水线（代码在 `xiaozhi-server/core/providers/`，每个环节都是可插拔的 Provider）：

```
OPUS音频 → 解码PCM → VAD(有人说话吗?) → ASR(说了什么? 语音→文字)
  → Intent(意图识别: 闲聊?点歌?控制设备?) 
  → Memory(取出相关记忆) + Prompt(人格设定) 
  → LLM(大模型生成回复, 可能调用工具/MCP)
  → TTS(文字→语音, 流式) → OPUS编码 → 发回设备
```

### 各环节详解

| 环节 | 全称 | 作用 | 本项目可选供应商举例 |
|---|---|---|---|
| VAD | Voice Activity Detection | 检测语音起止，判断"说完了" | SileroVAD（本地） |
| ASR | Automatic Speech Recognition | 语音转文字 | FunASR（本地免费）、豆包、阿里云 |
| LLM | Large Language Model | 理解与生成回复 | Qwen、DeepSeek、ChatGLM、豆包… |
| VLLM | Vision LLM | 看图（摄像头视觉分析） | Qwen-VL 等 |
| TTS | Text To Speech | 文字转语音 | EdgeTTS（免费）、豆包TTS、CosyVoice（可克隆音色） |
| Intent | 意图识别 | 决定走闲聊还是调用功能 | LLM 判断 / function_call |
| Memory | 记忆 | 跨对话记住用户信息 | 本地短期记忆 / mem0ai |

**延迟是语音交互的生命线**。总延迟 ≈ VAD 判停 + ASR + LLM 首 token + TTS 首包。优化手段：全链路流式（LLM 边生成、TTS 边合成、边发回设备）、选低延迟供应商（用 `performance_tester.py` 实测）、开启唤醒词回复缓存。

## 2. 大模型（LLM）核心知识

### 2.1 基本原理认知

- LLM 本质是"预测下一个 token"的模型；**它没有内置记忆**，每次请求都要把系统提示词+历史对话完整发过去。
- **上下文窗口**：一次能塞多少 token（几万~几十万）。对话太长要裁剪或摘要——这就是 Memory 模块存在的原因。
- **温度（temperature）**：越高越有创造性/随机，越低越稳定。陪伴闲聊可 0.7~0.9，工具调用宜低。
- **幻觉**：模型会一本正经编造事实。靠提示词约束 + 知识库（RAG）缓解。

### 2.2 提示词工程（Prompt Engineering）——人格的来源

小智的人格 = 系统提示词。看 `xiaozhi-server/agent-base-prompt.txt`，它定义了基础行为框架；智控台里每个"智能体"的角色设定会拼接进去。

写好陪伴机器人提示词的要点：

1. **身份**：名字、性格、说话风格（"你是XX，说话简短温暖，偶尔调皮"）
2. **口语化约束**：回复会被 TTS 朗读——要求"不用列表、不用 markdown、每次不超过两三句"
3. **边界**：不懂就说不懂；涉及健康/安全给出谨慎建议
4. **记忆引用**：如何自然地使用记忆里的信息（"记得用户上次说……"）
5. 迭代方法：每次只改一处 → 用固定的几句测试话术对比效果

### 2.3 工具调用与 MCP

LLM 本身只会生成文字，**工具调用（function calling）**让它能"做事"：把工具列表（名字+描述+参数 schema）给模型，模型决定何时输出一个结构化调用请求。

**MCP（Model Context Protocol）**是 Anthropic 提出的开放协议，把"工具"标准化成可插拔的服务。小智生态里有三层工具：

```
① 设备端 MCP（固件 mcp_server.cc）：音量、灯、电机、GPIO —— 控制硬件
② 服务器插件（plugins_func/functions/）：天气、新闻、点歌、Home Assistant
③ 云端 MCP 接入点（mcp_server_settings.json）：接入任意外部 MCP 服务
```

一次"帮我把灯调暗"的旅程：ASR→LLM 看到设备上报的工具列表→生成调用 `self.lamp.set_brightness(30)`→服务器经 WebSocket 转发给设备→设备执行→结果回传→LLM 生成"好啦，调暗了"→TTS。

### 2.4 记忆与知识库（RAG）

- **记忆（Memory）**：把对话中的关键信息（喜好、名字、事件）摘要存储，下轮对话检索注入提示词 → 陪伴感的核心。
- **RAG（检索增强生成）**：把文档切块→向量化→存入向量库；提问时检索最相关片段塞给 LLM。本项目通过 `search_from_ragflow.py` 插件对接 RAGFlow 知识库。
- 区别：记忆是"关于用户的"，知识库是"关于世界/领域的"。

## 3. 陪伴机器人的 AI 设计要点

1. **人格一致性**：所有模块（提示词、TTS 音色、表情、开场白）服务同一个人设。
2. **主动性**：好的陪伴不只是有问必答——定时问候、根据记忆主动关心（可用插件+定时任务实现）。
3. **情绪表达**：LLM 输出情绪标签 → 设备显示对应表情（固件 `llm` 消息里的 emotion 字段）。
4. **声纹识别**：区分家庭成员，对不同人用不同称呼和记忆。
5. **安全底线**：对老人/儿童场景，提示词中明确医疗、危险行为的应答边界。

## 4. 动手实验（过关标准）

1. **换模型对比**：`data/.config.yaml` 里分别切换两个 LLM（如 Qwen vs DeepSeek），用同样 5 个问题对比回复质量与延迟（`performance_tester.py` 测延迟）。
2. **人格改造**：写一个全新角色提示词（比如"东北话唠嗑大爷"），验证说话风格明显改变。
3. **读一个插件源码**：精读 `plugins_func/functions/get_weather.py`，画出"用户问天气→LLM→插件→回复"的调用链。
4. **写第一个自定义插件**：仿照 `get_time.py` 写一个"报今天日程"的假数据插件，注册后语音验证 LLM 会调用它。
5. **记忆验证**：告诉机器人你的生日，断开重连后问它是否记得，观察 Memory 模块日志。
